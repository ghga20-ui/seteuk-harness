# 불변 원칙
- **이 도구는 평가를 대신하지 않는다. 교사의 평가 자료(채점표, 루브릭 점수, 관찰 기록 등)가 확인되지 않은 활동은 세특을 생성하지 않는다** — 채점 완료를 먼저 안내하고 중단한다(검증기도 평가자료 미기록 시 저장을 차단한다)
- **톤(수준) 판정의 주권은 교사에게 있다.** 톤 등급은 교사의 평가 자료에서 기계적으로 파생하는 것을 기본으로 하고, 원문 품질에 따른 조정이 필요해 보이면 생성 전에 조정안을 질문으로 제시해 승인된 것만 반영한다. AI가 조정해 놓고 사후 통보하지 않는다
- 학생 원본에 실제로 있는 내용만 쓴다. 미화·창작 금지
- 검증 FAIL 상태에서는 어떤 산출물도 저장하지 않는다
- 위키(활동이력 포함)에 학생 실명·학번·원문을 기록하지 않는다
- 위키 개정은 사용자 승인 후에만 한다
- 초안은 교사 검수·승인 전에는 NEIS 입력 후보가 아니다. 이를 사용자에게 명확히 안내한다
- 모든 응답은 한국어로 한다

# 개입 지도 (사람이 결정하는 곳)
이 하네스는 Human in the Loop 원칙으로 설계되었다. 사람은 판단이 필요한 고판단·저빈도 지점에, 기계는 저판단·고빈도 지점에 배치한다. 학생별 개별 승인처럼 사람을 도장 찍는 기계로 만드는 설계는 하지 않는다.

| 단계 | 사람의 역할 | 방어하는 실패 |
|---|---|---|
| 채점 | **결정(In)** — 평가 없이는 시작 불가 | 평가 행위의 AI 위임 |
| 규칙·구조 인터뷰 | **결정(In)** — 선택지 비교로 확정 | 획일적 기본값 |
| 파일럿 배치 | **결정(In)** — 표본 승인 후 배치 생성 | 배치 전체의 문체·톤 오류 |
| 배치 생성·기계 검증 | 자동(Out) — 규칙은 사람이 정한 것 | 단순 반복의 사람 소모 |
| 검수 | **감독(On)** — 우선 검토 대상부터 | 균등 검토의 도장 찍기화 |
| 회고·확정 | **결정(In)** — 패턴별 승인, 확정 선언 | AI의 자가 학습 폭주 |

# 실행 전 점검 (환경 자동 준비)
사용자에게 설치를 미루지 말고 스킬이 직접 환경을 준비한다. 사용자는 권한 승인만 하면 된다.

1. 파이썬 탐지: `python --version`이 실패하면 `py --version`, `python3 --version`을 차례로 시도한다. 성공한 명령을 이 세션의 파이썬 명령으로 삼는다(이후 모든 스크립트 실행에 사용)
2. 파이썬이 없으면 사용자에게 한 줄로 알리고 직접 설치한다:
   - Windows: `winget install -e --id Python.Python.3.12` (winget이 없으면 Microsoft Store의 Python 설치를 안내)
   - macOS: `brew install python3` (brew가 없으면 python.org 설치 파일 안내)
   - 설치 후 새 셸에서 다시 탐지해 경로 반영을 확인한다
3. openpyxl 확인: `<파이썬> -c "import openpyxl"` — 실패하면 `<파이썬> -m pip install openpyxl`을 직접 실행한다(pip가 없으면 `<파이썬> -m ensurepip --upgrade` 먼저)
4. 자동 설치까지 실패한 경우에만(프록시 차단 등) 수동 해결 방법을 안내하고 중단한다. 환경이 준비되기 전에는 생성을 시작하지 않는다
5. 대상이 3개 학급 이상이면 컨텍스트 한도를 고려해 반 단위로 나눠 실행하는 흐름을 기본으로 제안한다
6. 이 스킬은 학생의 글·산출물 텍스트가 있는 활동 전용이다. 실기 영상·연주처럼 텍스트 산출물이 없는 활동이면 범위 밖임을 알리고 중단한다

# 파이프라인 (순서를 바꾸지 않는다)

## 1. 입력 파악 (입력 정규화)
학생 원본이 어떤 포맷으로 오든 이후 단계가 포맷을 몰라도 되도록 학생별 텍스트로 정규화한다. 특정 MCP나 외부 도구 설치를 전제하지 않는다.

1. 인벤토리: 원본 파일을 나열하고 확장자가 아니라 파일 헤더로 실제 포맷을 판정한다(번들 추출기가 판정과 추출을 함께 수행)
2. 포맷 라우팅 — 4단 폴백을 순서대로 시도한다:
   1) 세션에 한글 문서 파싱 도구(MCP·스킬)가 있으면 활용한다(선택 가속기일 뿐 필수가 아님)
   2) 번들 추출기를 사용한다: `python <스킬경로>/scripts/extract_sources.py <파일>` — hwpx, xlsx, 확장자 위장 HTML, 일반 텍스트 지원
   3) 사진(jpg, png)과 스캔 pdf는 파일을 직접 읽어 비전으로 전사한다(외부 OCR 불필요). 손글씨 전사는 학생별 신뢰도를 기록하고, 판독이 안 되는 구간은 지어내지 않고 [판독불가]로 남기며 그 구간을 세특 근거로 쓰지 않는다
   4) 위 방법이 모두 실패한 포맷(hwp 바이너리 등)은 사용자에게 한글에서 hwpx나 pdf로 다시 저장해 달라고 안내한다
3. 고유명사 즉시 대조: 작품명 등 흔들리기 쉬운 고유명사는 전사 직후 활동 참조 목록과 대조하고 불일치는 확인 목록에 올린다
4. 명렬 확보: 전체 수강생의 학번·이름 목록을 확보한다(명렬 파일이 있으면 파싱, 없으면 사용자에게 확인). `명렬.json`으로 저장한다: `{"students": [{"학번": "10101", "이름": "김가상"}]}` — 명렬 없이는 미제출자를 감지할 수 없음을 사용자에게 알린다
5. 학생 매핑: 파일명·시트·표 구조에서 학번과 이름을 추출하고 **학번과 이름을 함께 명렬과 대조한다**(크로스체크). 하나라도 불일치하면 오전사·오매핑 의심으로 확인 목록에 올린다. 잘못된 학생에게 세특이 귀속되는 것이 이 도구의 최악의 실패이므로 추정 배정을 절대 하지 않는다
6. 활동 안내문이 있으면 같은 방식으로 파싱해 활동명·평가 요소·참조 목록을 확보한다
7. 표본 학생 2~3명(수준이 달라 보이는)을 골라 원문을 읽고 활동 성격을 파악한다

## 2. 규칙 인터뷰 (질문은 반드시 하나씩)
| # | 질문 | 기본값 |
|---|---|---|
| 1 | 활동명과 문두 형식(`<활동명>에서` / `<활동명>을 통해`) | 없음(필수) |
| 2 | **교사 평가 자료의 위치**(채점표, 루브릭 점수, 관찰 기록) | **없음(필수) — 없으면 채점 완료를 안내하고 여기서 중단** |
| 3 | 목표 분량 | 700바이트(상한 760) |
| 4 | 활동 특수 참조자료(소재 교정 기준 목록 등) | 안내문에서 추출 후 확인 |
| 5 | 톤 차등(단계 수·점수 구간 매핑) | 교사 평가 점수에서 파생. AI의 조정 의견은 생성 전 질문으로 |
| 6 | 예외 처리(빈 답안, 짧은 답안, 목록 외 소재) | 빈=참여 사실만, 짧음=짧게, 목록 외=확인 요청 |
| 7 | 학생별 교사 관찰 메모(선택) | 없어도 진행. 있으면 해당 학생 세특에 결합 — 산출물에 없는 교사만의 관찰이 세특을 기록답게 만든다 |

- 위키에 같은 답이 있으면 "지난번과 같이 갈까요?"로 축소한다
- 확정값을 수행평가 폴더의 `활동프로파일.json`에 기록한다. 최소 키: `활동명`, `문두`, `목표바이트`, `상한바이트`, `평가자료`(출처와 확인 일자), `톤단계`, `예외방침`, `채택구조` — `평가자료`가 비어 있으면 검증기가 저장을 차단한다

## 3. 구조 심층 인터뷰 (하네스의 심장)
1. 표본 분석과 활동 성격을 근거로 [wiki/구조패턴.md](wiki/구조패턴.md)에서 후보 2~3개를 고른다(유사 회차 채택 구조를 1안으로)
2. **같은 표본 학생 1명의 원문으로 후보별 완성 세특을 만들어 나란히 시연한다**
3. 사용자가 선택하면 "고칠 요소가 있나요?" 조정 질문을 1회 한다
4. 확정 구조(패턴 ID + 조정 내역)를 활동프로파일.json에 기록한다

## 4. 파일럿 배치 (생성 구간의 사람 개입 지점)
전체 생성 전에 소량으로 실전 문체를 확정받는다. 구조 시연(구조 고르기)과 목적이 다르다 — 여기서는 확정된 구조로 쓴 실제 초안의 문체·톤 수위·개요 방식을 확인받는다.

1. 톤 등급별 표본 3~4명(예: 상·중·중하 각 1명)의 세특을 먼저 완성한다
2. 완성본을 사용자에게 나란히 보여 주고 고칠 점을 확인받는다
3. 지적이 나오면 파일럿에 반영해 재확인받고, 확정된 방식을 나머지 전원에게 적용한다
4. **파일럿 승인 없이 배치 생성을 시작하지 않는다** — 배치 후 발견된 문체 오류는 전원 재작성 비용이 된다

## 5. 생성 루프
학생별로: 원문 읽기 → 개별 강점 추출 → 톤 판정 → 확정 구조와 [wiki/표현사전.md](wiki/표현사전.md)·[wiki/문체관찰.md](wiki/문체관찰.md)를 적용해 작성.
- 소재가 참조 목록과 불일치하면: 내용 근거로 교정을 제안하고 사용자 확인 후 비고에 기록
- 예외 학생은 인터뷰에서 확정한 방침대로 처리하고 `"예외": true`로 표시
- 서로 다른 학생의 원본이 실질적으로 동일하면(표절 의심) 세특 작성과 별개의 평가 사안이므로 교사에게 보고한다

## 6. 검증 루프 (2단)
1. 초안을 `초안.json`으로 만들고 결정론 채점기를 실행한다. 명렬이 있으면 `--roster 명렬.json`을 반드시 붙인다(학번·이름 대조와 미제출자 감지):

초안.json 계약(키 이름을 그대로 사용):
```json
{"classes": [{"name": "1반", "students": [{"학번": "10101", "이름": "김가상", "핵심소재": "가상의 책(작가)", "톤등급": "중", "세특": "...", "비고": "", "예외": false}]}]}
```
예외 학생(빈 답안 등)은 "예외": true로 표시한다.
   ```
   python <스킬경로>/scripts/verify_seteuk.py 초안.json --profile 활동프로파일.json --roster 명렬.json
   ```
2. FAIL이 있으면 해당 학생만 수정하고 재실행한다(수정 시 원문 근거는 삭제하지 않고 수식어부터 줄인다)
3. 결정론 통과 후 [wiki/루브릭.md](wiki/루브릭.md) 기준으로 학생별 LLM 채점을 한다(생성과 분리된 검증 관점으로, 원본과 세특을 대조)
4. 루브릭 FAIL은 사유를 피드백으로 재작성한다. **학생당 재작성 상한 3회**, 초과 시 비고에 '검토 필요'를 기록하고 다음 학생으로 진행한다

## 7. 저장 게이트
전원 처리 후 `--save`로 재검증과 저장을 한 번에 실행한다:
```
python <스킬경로>/scripts/verify_seteuk.py 초안.json --profile 활동프로파일.json --save "세특(초안).xlsx"
```
FAIL이 하나라도 있으면 스크립트가 저장을 차단한다. 우회 저장을 하지 않는다.
저장되는 파일의 첫 시트에는 `안내` 시트(AI 초안 — 교사 검수 전, NEIS 입력 금지)가 자동 포함된다. 이 표시는 회고(검수 완료) 전까지 제거하지 않는다.
저장 성공 직후 같은 폴더에 검수용 사본을 자동 생성한다: 파일명의 `(초안)`을 `(검수용)`으로 바꾼 복사본. 초안 파일은 회고 diff의 기준선이므로 수정하지 않고 보존한다.
사본의 각 시트 마지막 열에 '검수 메모' 열을 추가한다. 선택 입력 채널이다 — 사용자가 수정 이유를 한마디 적으면 회고가 의도를 추론하지 않고 그대로 학습하고, 비워 두면 diff에서 추론한다.

## 8. 검수 보고
- **우선 검토 대상을 이유와 함께 명시한다**: 전사 신뢰도 낮음, 톤 조정 학생, 루브릭 재작성 발생, 소재 교정, 답안 끊김·짧음 등 판단이 필요한 학생을 먼저 제시하고, 나머지는 표본 검토를 안내한다. 전건 균등 검토는 전건 대충 검토가 되기 쉬우므로 교사의 주의를 판단이 필요한 곳에 배분한다
- 확인 필요 목록(빈 답안, 소재 교정, 추정, 검토 필요)을 절대 생략하지 않고 보고한다
- **미제출자 명단을 반드시 포함한다** — 원본이 없어 세특이 생성되지 않은 학생을 조용히 누락시키지 않는다
- 초안은 교사 검수·승인 전에는 NEIS에 입력하면 안 된다는 점을 보고에 명시한다(허위·과장·기재요령 준수의 최종 확인 책임은 교사에게 있음)
- 사용자에게 수정은 반드시 검수용 사본에서만 하도록 안내한다(초안은 비교 기준선)
- 학급이 여러 개면 첫 학급 검수 후 나머지를 진행하는 흐름을 기본으로 제안한다

## 9. 회고 (사용자가 검수 수정본을 주면)
1. 초안 파일의 수정 시각을 확인한다. 저장 게이트 실행 이후 초안이 수정된 흔적이 있으면 기준선 오염 가능성을 사용자에게 경고하고 어느 파일이 기준인지 확인한다
2. 초안과 검수용을 행 순서가 아니라 학번 키로 매칭해 diff를 만든다(사용자가 행을 정렬·삭제해도 비교가 깨지지 않게)
3. **무수정 감지**: diff가 0건이면 초안이 완벽했을 가능성과 검수가 충분하지 않았을 가능성을 함께 짚고, 우선 검토 대상을 확인하셨는지 물은 뒤 진행한다(비난이 아니라 확인 — 도장 찍기 방지 장치)
4. 검수 메모가 적힌 행은 메모를 수정 의도로 그대로 학습하고, 메모가 없는 행만 diff에서 반복 패턴을 추론한다
5. 패턴별로 "규칙으로 반영할까요?"를 묻는다
6. 승인된 것만 기록한다: 문체 경향→[wiki/문체관찰.md](wiki/문체관찰.md), 표현 승격·강등→[wiki/표현사전.md](wiki/표현사전.md), 규칙 개정→[wiki/규칙정본.md](wiki/규칙정본.md)(개정 이력 포함), 검수 지적 유형→[wiki/루브릭.md](wiki/루브릭.md). **금지 어휘·특수문자·분량 규칙을 개정할 때는 기계 정본인 `wiki/규칙.json`을 반드시 함께 갱신한다**(검증기는 이 파일을 읽는다)
7. 이번 회차를 `wiki/활동이력/YYYY-MM-활동명.md`로 기록한다(`_템플릿.md` 형식, 익명 통계만)
8. [wiki/구조패턴.md](wiki/구조패턴.md)의 채택 패턴에 사용 실적을 추가한다
9. 회고를 마친 활동에 한해, 사용자가 요청하면 확정본을 만든다: 검수 반영본에서 `안내` 시트를 제거하고 파일명을 `(확정)`으로 저장. **회고를 거치지 않은 활동의 확정본은 만들지 않는다** — 확정은 교사의 검수가 완료되었다는 선언이다
