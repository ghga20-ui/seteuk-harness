# 불변 원칙
- 학생 원본에 실제로 있는 내용만 쓴다. 미화·창작 금지
- 검증 FAIL 상태에서는 어떤 산출물도 저장하지 않는다
- 위키(활동이력 포함)에 학생 실명·학번·원문을 기록하지 않는다
- 위키 개정은 사용자 승인 후에만 한다
- 모든 응답은 한국어로 한다

# 위키 선독 (매 실행 시작)
다음을 전부 읽는다: [wiki/규칙정본.md](wiki/규칙정본.md), [wiki/구조패턴.md](wiki/구조패턴.md), [wiki/표현사전.md](wiki/표현사전.md), [wiki/루브릭.md](wiki/루브릭.md), [wiki/문체관찰.md](wiki/문체관찰.md), `wiki/활동이력/`의 최근 3개 파일.
활동이력에서 이번 활동과 유사한 회차를 찾으면 그 결정(구조, 톤, 예외 방침)을 이번 인터뷰의 1안으로 삼는다.

# 파이프라인 (순서를 바꾸지 않는다)

## 1. 입력 파악
1. 학생 원본을 파싱한다(kordoc 우선, 실패 시 파일 헤더를 확인해 HTML 위장 xls 등 폴백 파서 작성)
2. 활동 안내문이 있으면 파싱해 활동명·평가 요소·참조 목록을 확보한다
3. 표본 학생 2~3명(수준이 달라 보이는)을 골라 원문을 읽고 활동 성격을 파악한다

## 2. 규칙 인터뷰 (질문은 반드시 하나씩)
| # | 질문 | 기본값 |
|---|---|---|
| 1 | 활동명과 문두 형식(`<활동명>에서` / `<활동명>을 통해`) | 없음(필수) |
| 2 | 목표 분량 | 700바이트(상한 760) |
| 3 | 활동 특수 참조자료(소재 교정 기준 목록 등) | 안내문에서 추출 후 확인 |
| 4 | 톤 차등(여부·단계·수위) | 3단계 상/중/하 |
| 5 | 예외 처리(빈 답안, 짧은 답안, 목록 외 소재) | 빈=참여 사실만, 짧음=짧게, 목록 외=확인 요청 |

- 위키에 같은 답이 있으면 "지난번과 같이 갈까요?"로 축소한다
- 확정값을 수행평가 폴더의 `활동프로파일.json`에 기록한다. 최소 키: `활동명`, `문두`, `목표바이트`, `상한바이트`, `톤단계`, `예외방침`, `채택구조`

## 3. 구조 심층 인터뷰 (하네스의 심장)
1. 표본 분석과 활동 성격을 근거로 [wiki/구조패턴.md](wiki/구조패턴.md)에서 후보 2~3개를 고른다(유사 회차 채택 구조를 1안으로)
2. **같은 표본 학생 1명의 원문으로 후보별 완성 세특을 만들어 나란히 시연한다**
3. 사용자가 선택하면 "고칠 요소가 있나요?" 조정 질문을 1회 한다
4. 확정 구조(패턴 ID + 조정 내역)를 활동프로파일.json에 기록한다

## 4. 생성 루프
학생별로: 원문 읽기 → 개별 강점 추출 → 톤 판정 → 확정 구조와 [wiki/표현사전.md](wiki/표현사전.md)·[wiki/문체관찰.md](wiki/문체관찰.md)를 적용해 작성.
- 소재가 참조 목록과 불일치하면: 내용 근거로 교정을 제안하고 사용자 확인 후 비고에 기록
- 예외 학생은 인터뷰에서 확정한 방침대로 처리하고 `"예외": true`로 표시

## 5. 검증 루프 (2단)
1. 초안을 `초안.json`으로 만들고 결정론 채점기를 실행한다:
   ```
   python <스킬경로>/scripts/verify_seteuk.py 초안.json --profile 활동프로파일.json
   ```
2. FAIL이 있으면 해당 학생만 수정하고 재실행한다(수정 시 원문 근거는 삭제하지 않고 수식어부터 줄인다)
3. 결정론 통과 후 [wiki/루브릭.md](wiki/루브릭.md) 기준으로 학생별 LLM 채점을 한다(생성과 분리된 검증 관점으로, 원본과 세특을 대조)
4. 루브릭 FAIL은 사유를 피드백으로 재작성한다. **학생당 재작성 상한 3회**, 초과 시 비고에 '검토 필요'를 기록하고 다음 학생으로 진행한다

## 6. 저장 게이트
전원 처리 후 `--save`로 재검증과 저장을 한 번에 실행한다:
```
python <스킬경로>/scripts/verify_seteuk.py 초안.json --profile 활동프로파일.json --save "세특(초안).xlsx"
```
FAIL이 하나라도 있으면 스크립트가 저장을 차단한다. 우회 저장을 하지 않는다.

## 7. 검수 보고
- 확인 필요 목록(빈 답안, 소재 교정, 추정, 검토 필요)을 절대 생략하지 않고 보고한다
- 학급이 여러 개면 첫 학급 검수 후 나머지를 진행하는 흐름을 기본으로 제안한다

## 8. 회고 (사용자가 검수 수정본을 주면)
1. 초안과 수정본의 diff에서 반복 패턴을 찾는다
2. 패턴별로 "규칙으로 반영할까요?"를 묻는다
3. 승인된 것만 기록한다: 문체 경향→[wiki/문체관찰.md](wiki/문체관찰.md), 표현 승격·강등→[wiki/표현사전.md](wiki/표현사전.md), 규칙 개정→[wiki/규칙정본.md](wiki/규칙정본.md)(개정 이력 포함), 검수 지적 유형→[wiki/루브릭.md](wiki/루브릭.md)
4. 이번 회차를 `wiki/활동이력/YYYY-MM-활동명.md`로 기록한다(`_템플릿.md` 형식, 익명 통계만)
5. [wiki/구조패턴.md](wiki/구조패턴.md)의 채택 패턴에 사용 실적을 추가한다

# 질문 규율
- 묻지 않는다: 파일에서 파생되는 값, 규칙정본에 있는 것, 위키로 답할 수 있는 것(확인만)
- 묻는다: 인터뷰 5문항, 구조 시연 선택, 원본 모호성, 3회 검증 실패 처리, 회고 반영 승인

# 완료 조건
- 전원 결정론 검증 PASS(예외·검토 필요는 비고에 명시)
- xlsx 저장 완료
- 확인 필요 목록 보고 완료
